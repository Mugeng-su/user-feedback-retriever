# V1 功能清单与实现方案

## 1. 文档目的

这份文档承接 PRD，目标是把产品想法进一步收敛成一个可以进入设计和开发阶段的 V1 方案。

重点回答 3 个问题：

1. 第一版输入形式到底是什么
2. 第一版页面要展示哪些模块
3. 第一版技术路线应该怎么选

## 2. V1 设计原则

第一版不是为了覆盖所有可能场景，而是为了尽快验证一个最小闭环：

`团队提交一条高表现视频及其评论 -> 系统输出一份可直接用于复盘和下次创作的结构化分析报告`

因此 V1 需要遵循以下原则：

- 输入尽量简单，优先手动导入
- 输出必须结构化，不能只是聊天回答
- 分析必须围绕“下一步可执行”
- 先支持单条视频复盘，不做复杂对比分析
- 先把分析质量做好，再考虑数据抓取自动化

## 3. V1 目标用户

### 3.1 第一优先用户
- 内容运营
- 品牌营销
- KOL 合作运营

### 3.2 他们最核心的任务
- 复盘一条高表现视频
- 找到可复用的表达和结构
- 提炼用户真实语言
- 帮下一条内容创作提供输入

## 4. V1 版本定位

### 4.1 产品定位
一个内部使用的 AI 内容复盘工作台。

### 4.2 不把自己定义成什么
- 不定义成全网爆款分析平台
- 不定义成自动抓取所有平台数据的爬虫系统
- 不定义成直接生成视频的 AI 创作工具

## 5. V1 核心任务

V1 只需要把下面这件事做好：

`给定一条视频的基础信息、文稿和评论，输出一份高质量复盘报告。`

这份报告必须回答：

1. 这条视频为什么有效
2. 评论区用户最在意什么
3. 用户如何表达这个产品价值
4. 下一条内容应该怎么做

## 6. V1 输入方案

### 6.1 推荐方案：手动导入
第一版建议采用手动导入，而不是一开始就接平台抓取。

原因：
- 开发复杂度最低
- 可绕开平台 API 和抓取限制
- 更快验证“分析报告是否有价值”
- 更适合内部试用和快速迭代

### 6.2 V1 必填输入

建议 V1 至少支持以下字段：

- 视频标题
- 视频链接
- 平台名称
- 播放量/点赞量/评论量/分享量（可选部分也可以）
- 视频文稿或字幕文本
- 评论列表

### 6.3 评论输入格式建议
建议支持以下两种方式之一：

1. 直接粘贴原始评论，一行一条
2. 上传 CSV 文件

V1 优先推荐第一种，最快可用。

### 6.4 暂不做的输入能力
- 自动抓取评论
- 自动下载视频并转写
- 批量导入多条视频

这些可以放到 V1.1 或 V2。

## 7. V1 输出方案

V1 输出不应是聊天式回答，而应是一份固定结构的分析结果页。

### 7.1 输出模块

建议包含以下模块：

1. `总览摘要`
2. `视频爆点拆解`
3. `视频结构复盘`
4. `评论主题分类`
5. `用户原话摘录`
6. `用户语言与卖点表达`
7. `下一条内容建议`

### 7.2 每个模块的价值

`总览摘要`
- 用 3 到 5 条结论快速说明这条视频为什么值得复盘

`视频爆点拆解`
- 解释这条视频表现好的关键因素

`视频结构复盘`
- 说明开头、主体、展示方式、节奏等结构信息

`评论主题分类`
- 看清楚评论区主要在讨论什么

`用户原话摘录`
- 保留用户真实表达，方便直接复用

`用户语言与卖点表达`
- 把评论转成适合文案、脚本、SEO 的表达资产

`下一条内容建议`
- 给内容团队直接可执行的建议

## 8. V1 页面结构

## 8.1 页面数量建议
第一版建议只做 3 个核心页面：

1. `首页 / 新建分析`
2. `分析处理中`
3. `分析结果页`

### 8.2 页面一：首页 / 新建分析
页面目标：让用户快速提交一条视频分析任务。

建议包含：
- 产品标题与用途说明
- 视频基础信息输入区
- 视频文稿输入区
- 评论输入区
- 提交分析按钮

### 8.3 页面二：分析处理中
页面目标：让用户知道系统正在做什么，减少等待焦虑。

建议展示：
- 当前处理状态
- 正在执行的步骤
- 预计等待说明

步骤文案可参考：
- 正在解析视频文稿
- 正在识别视频核心结构
- 正在整理评论主题
- 正在提炼用户语言
- 正在生成复盘建议

### 8.4 页面三：分析结果页
页面目标：清晰展示一份完整的复盘报告。

建议布局：
- 顶部：视频信息与关键指标
- 中部：视频拆解、评论分析、用户语言
- 底部：下一条内容建议和可复用表达

## 9. V1 功能清单

### 9.1 必做功能
- 创建单条分析任务
- 保存输入内容
- 调用 AI 完成结构化分析
- 展示结构化结果页
- 支持复制分析结果中的重点内容

### 9.2 可选但推荐功能
- 支持导出 Markdown 报告
- 支持重新生成某个模块
- 支持标签化评论分类结果

### 9.3 暂不做功能
- 多人协作
- 权限管理
- 历史报告高级筛选
- 跨视频对比分析
- 自动抓取平台数据

## 10. AI 输出结构建议

为了保证输出稳定，建议后端不要直接请求一段自由文本，而是让模型按固定 JSON 结构返回结果。

### 10.1 建议的结构

```json
{
  "summary": [],
  "video_hooks": [],
  "video_structure": {
    "hook": "",
    "problem": "",
    "solution": "",
    "proof": "",
    "cta": ""
  },
  "viral_factors": [],
  "comment_topics": [
    {
      "topic": "",
      "count_estimate": "",
      "sentiment": "",
      "examples": []
    }
  ],
  "user_phrases": [],
  "value_points": [],
  "pain_points": [],
  "copywriting_angles": [],
  "next_content_suggestions": []
}
```

### 10.2 这样做的原因
- 前端更容易稳定渲染
- 方便后续做导出和知识库沉淀
- 更容易做局部重生成
- 便于未来做多条内容对比

## 11. 技术路线建议

## 11.1 推荐路线
V1 建议采用：

`手动录入 + LLM 结构化分析 + Web 结果页展示`

### 11.2 这样选的原因
- 开发速度快
- 风险低
- 最适合先验证产品价值
- 可以先不碰复杂的数据采集问题

## 11.3 技术模块拆分

建议拆成 4 层：

1. `输入层`
负责接收视频信息、文稿和评论

2. `预处理层`
负责清洗评论、去空行、去重、截断超长文本

3. `AI 分析层`
调用模型生成结构化分析结果

4. `展示层`
把分析结果渲染成报告页面

## 12. 建议的开发顺序

### 12.1 第一步
先做静态页面和输入表单。

### 12.2 第二步
定义 AI 返回的 JSON schema，并打通最小分析链路。

### 12.3 第三步
完成结果页模块渲染。

### 12.4 第四步
补充导出、重新分析和基础历史记录。

## 13. 数据模型建议

V1 可以先用一个非常简单的数据结构。

### 13.1 Analysis
- id
- title
- platform
- video_url
- metrics
- transcript
- comments_raw
- status
- result_json
- created_at

### 13.2 Comment
如果 V1 想简化，也可以不单独建表，先直接存原始文本。

如果后面要做评论筛选和标签功能，再拆独立结构。

## 14. 关键产品取舍

### 14.1 为什么先不做自动抓取
因为自动抓取解决的是“拿数据更方便”，但第一阶段更该验证的是“分析结果是否真有价值”。

### 14.2 为什么先不做多视频分析
因为单条复盘已经足够验证核心价值，多条对比会显著增加页面和算法复杂度。

### 14.3 为什么强调用户原话
因为这部分是和通用分析工具拉开差距的关键，也是最容易进入内容、文案和 SEO 工作流的资产。

## 15. V1 成功标准

V1 上线后，可以用下面几个标准判断是否值得继续：

- 用户是否愿意反复提交不同视频分析
- 用户是否会复制和使用“用户原话”与“下一条建议”
- 团队是否会把结果带进复盘会或脚本会
- 用户是否觉得这比人工复盘快且更有启发

## 16. 我对 V1 的最终建议

第一版就做成一个`内部 AI 复盘工作台`最合适。

它不是为了把所有问题一次解决，而是为了先把最有价值的一小段闭环跑通：

`复盘高表现视频 -> 读懂评论区 -> 提炼用户语言 -> 指导下一条内容`

只要这条链路成立，后面无论是扩到竞品分析、行业趋势、文案生成还是 SEO 语料库，都会顺很多。
